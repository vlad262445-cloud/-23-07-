# purchase_vendor_card

Модуль из раздела 8 ТЗ (модуль 7 в нумерации ТЗ), восьмой по очереди
внедрения (п. 11.1) - полноценная финансовая картина по поставщику прямо
на карточке `res.partner`: выставлено/оплачено/едет/должны, оборот за
период, плюс качество данных и поиск дублей.

## Что делает

### Финансовые показатели (п. 8.2/8.3) - НЕ хранятся, считаются на лету

`vendor_invoiced_total`, `vendor_paid_total`, `vendor_frozen_total`
("оплачено и едет"), `vendor_debt_total` ("мы должны"),
`vendor_residual_total`, `vendor_order_count`, `vendor_avg_order`,
`vendor_last_order_date`, `vendor_missing_docs_count`,
`vendor_turnover_period`. Формулы - те же, что в `purchase.finance.report`
(модуль 6) - тест `test_stats_match_finance_report_module6` проверяет их
совпадение на одном наборе данных.

Одна пара запросов (`purchase.order.search_read` + `stock.picking.search`)
на весь `recordset`, а не по одному на партнёра - `test_query_count_does_
not_scale_with_partner_count` проверяет это на 5 vs 25 поставщиках.
"Товар получен" - через проведённую входящую приёмку (`stock.picking`),
не через `request_state` (см. `NOTES.md` - тот же класс бага, что уже
описан и покрыт тестом в `purchase_finance_dashboard`).

### Карточка поставщика (п. 8.4)

Всё новое - только когда `supplier_rank > 0`. Смарт-кнопки «Заявки»,
«Платежи», «Мы должны» (видна только при долге). Вкладки: **Финансы**
(плитки + список заказов), **Что покупаем** (прайс-лист + кнопка «История
цен» - pivot по `purchase.order.line`), **Надёжность** (ссылка на отчёт
«Поставщики: срыв сроков» + индикатор качества данных), **Реквизиты**
(ИНН/КПП/банковские счета).

### Список поставщиков (п. 8.5)

Меню «Поставщики» в приложении «Реестр закупок». Фильтры «Есть долг»,
«Есть заморозка», «Нет ИНН», «Не заказывали > 90 дней» - несмотря на то,
что показатели не хранятся, фильтры реально работают (через `search=` на
полях, см. `NOTES.md`). Сортировка по денежным колонкам не работает - это
осознанный размен той же природы (п. 8.5 ТЗ прямо предупреждает об этом).

### Качество данных и дубли (п. 8.6)

`vendor_data_quality` (ok/incomplete/critical) - store=True, в отличие от
финансовых полей: это структурный индикатор, а не растущая касса.
`vendor_duplicate_group_key`/`vendor_is_possible_duplicate` - группировка
по нормализованному ИНН, иначе по нормализованному названию (без юр.
формы/кавычек/регистра). Меню «Найти дубли поставщиков» готовит выборку
и открывает список, сгруппированный по ключу - объединение делается уже
штатным мастером Odoo (`base.action_partner_merge`, привязан к списку
`res.partner` и появляется в меню «Действие» сам, свой мастер объединения
не пишем).

## Что не сделано и почему

- Правка мастеров импорта (искать сначала по ИНН) - не в этом модуле, это
  модуль 8 (`purchase_vendor_matching`), уже сделан раньше (п. 8.6 ТЗ).
- Своя модель для дублей/мастер слияния - не заводились, используется
  `base.partner.merge.automatic.wizard` как есть.
- "Надёжность": разбивка по срыву сроков - краткая сводка + ссылка на
  существующий отчёт `vendor.delay.report`, не дублируем его целиком.

## Тесты

`tests/test_vendor_stats.py` - совпадение с модулем 6, инвариант
"выставлено = оплачено + остаток", взаимоисключение заморозки/долга,
регрессия на `stock.picking`, поставщик без заказов даёт нули,
количество запросов не растёт с числом поставщиков, поиск дублей по
точному примеру из ТЗ (`ООО "Дата-В"` / `ООО Дата-В`).
