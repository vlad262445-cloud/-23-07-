# Заметки по реализации

## Финансовые поля - не хранятся, `@api.depends()` пустой

П. 8.3 ТЗ прямо требует "на лету, не хранить" (касса меняется каждой
оплатой/приёмкой - хранимое поле означало бы пересчёт при каждой строчке
`purchase.payment.line`/`stock.picking` во всей системе). `@api.depends()`
без аргументов - штатная идиома Odoo для "пересчитывать заново каждый раз,
хранимой цепочки зависимостей в принципе быть не может" (то же решение,
что и в `purchase_finance_workspace`/`purchase_finance_dashboard` для
аналогичных сумм). `vendor_turnover_period` - отдельный метод с
`@api.depends_context('vendor_stats_from', 'vendor_stats_to')`, а не
голый `@api.depends()`, т.к. значение реально зависит от `self.env.context`
(период приходит через контекст фильтра списка) - это тот случай, для
которого `depends_context` и предназначен, а не `depends`.

## `vendor_order_ids`/`vendor_supplierinfo_ids` - таких полей на `res.partner` не было

Предположение "у партнёра наверняка уже есть `purchase_order_ids`/
`seller_ids`" оказалось неверным - проверено `odoo shell`
(`res.partner._fields.get('purchase_order_ids')` → `None`, аналогично для
`seller_ids`). У `purchase.order` и `product.supplierinfo` есть реальный
Many2one `partner_id`, поэтому объявлены свои One2many-поля без всякого
compute - Odoo резолвит их через существующий инверс автоматически. Никакой
собственной логики агрегации тут не нужно, только объявление поля.

## Реквизиты в списке `purchase_order_line_extra_views.xml`-подобной вьюхе - убраны до установки

При написании `res_partner_views.xml` в список заказов на вкладке
«Финансы» изначально попали `property_purchase_currency_id` (поле, чьё
существование не было проверено) и `lifecycle_stage` (реально
принадлежит опциональному `purchase_registry_ux` - использовать его здесь
значило бы жёсткую зависимость от необязательного модуля). Оба убраны до
первой попытки установки - тот же класс ошибки, которого репозиторий уже
избегал раньше (проверять поле в шелле/исходнике перед использованием в
XML, а не полагаться на память о похожих модулях).

## Доступ ГД/Собственника - через `implied_ids`, без жёсткой ссылки на dashboard-модуль

Новые вкладки формы партнёра ограничены `groups="purchase.group_purchase_
user,purchase_pdf_import.group_chief_buyer,purchase_pdf_import.group_
accountant,purchase_pdf_import.group_observer"`. `group_observer`
принадлежит этому же модулю (`purchase_pdf_import`), но реальными его
участниками становятся и ГД/Собственник - `group_ceo`/`group_owner`
(заведены в `purchase_finance_dashboard`) объявлены с `implied_ids =
[group_observer]`, поэтому Odoo сам создаёт им членство в `group_observer`
при назначении роли. Это значит, что вкладки видны ГД/Собственнику без
единой строчки, ссылающейся на `purchase_finance_dashboard` напрямую - если
этот модуль когда-нибудь будет не установлен, форма партнёра не сломается
(просто не будет и самих ролей `group_ceo`/`group_owner`, что ожидаемо).

## Поиск по нехранимым полям - `search=`, а не хранимая копия

`vendor_frozen_total`/`vendor_debt_total`/`vendor_last_order_date` дают
рабочие фильтры в списке поставщиков (`filter_has_debt` и т.п.), несмотря
на то, что сами поля не хранятся - каждое объявлено с `search=
'_search_vendor_xxx'`, метод переводит `operator+value` в `[('id', 'in',
[...])]`, посчитанный ОДНИМ батч-проходом по всем поставщикам компании
(`_get_all_suppliers_stats`), а не по одному вызову на партнёра. Сортировка
по этим же колонкам в списке при этом не работает - Odoo умеет
транслировать `search`, но не `order`, для non-stored полей без `store=
True`; п. 8.5 ТЗ сам явно предупреждает об этом ограничении как о
принимаемом компромиссе, отдельно решать не требовалось.

`vendor_last_order_date` использует сентинел `1900-01-01` внутри
`_search_vendor_last_order_date` для поставщиков без единого заказа -
иначе фильтр "не заказывали > 90 дней" пропустил бы именно тех
поставщиков, у кого заказов вообще не было (самый крайний случай "давно не
заказывали").

## Качество данных и дубли - `store=True`, в отличие от финансовых полей

`vendor_data_quality`/`vendor_duplicate_group_key`/`vendor_is_possible_
duplicate` намеренно хранятся (обычный `@api.depends(...)` по реальным
полям контакта) - в отличие от финансов, это не растущая касса, а
структурный признак самой карточки контакта, меняющийся только при правке
реквизитов. Хранить его дешевле и логичнее, чем пересчитывать при каждом
открытии списка.

Известное ограничение (см. комментарий в коде): при смене ИНН только у
ОДНОГО партнёра из уже существующей группы-дубля соседние записи той же
группы не пересчитываются автоматически (их зависимость не включает поля
других записей). Мера - `action_open_vendor_duplicates` форс-пересчитывает
ВСЕХ поставщиков перед показом списка, поэтому экран "Найти дубли" всегда
показывает актуальную картину, даже если фоновые `store`-поля где-то
отстали.

## Тест на количество запросов - изолирован от чужих recompute через `flush_all()`

Первая версия `test_query_count_does_not_scale_with_partner_count` считала
лишние ~16 запросов на 20 доп. партнёров - оказалось, что тест ловил не
целевой расчёт (`vendor_invoiced_total` и т.п.), а отложенный пересчёт
ХРАНИМЫХ полей (`vendor_data_quality`, `vendor_duplicate_group_key`,
`vendor_is_possible_duplicate`), поставленный в очередь ещё при `create()`
в helper'е теста. Исправлено принудительным `self.env.flush_all()` +
`invalidate_recordset()` внутри `_make_n_suppliers_with_orders` ДО начала
окна замера - тест теперь проверяет ровно то свойство, которое требует
п. 8.3 ТЗ (не более O(1) запросов на финансовые поля), не задевая
несвязанную часть ORM.

## Тест на взаимоисключение заморозки/долга - сравнение с `amount_total`, не с сырыми суммами

Тот же нюанс, что уже был описан в `purchase_finance_dashboard/tests`: на
тестовом продукте может быть настроен налог поставщика по умолчанию,
из-за чего `order.amount_total` оказывается больше переданного
`price_unit`. Тест сравнивает `vendor_frozen_total`/`vendor_debt_total` с
`order.amount_total` соответствующего заказа, а не с жёстко прописанным
круглым числом - устойчиво к любым налоговым настройкам окружения.

## Что сознательно не реализовано

- **Правка мастеров импорта** (искать сначала по ИНН при создании
  поставщика) - это уже сделано раньше, в `purchase_vendor_matching`
  (п. 8.6 ТЗ, отдельный, уже развёрнутый модуль) - здесь не дублируется.
- **Собственная модель/мастер слияния дублей** - не заводились;
  используется штатный `base.action_partner_merge`
  (`binding_model_id=res.partner`, `binding_view_types=list`) - он сам
  появляется в меню «Действие» любого списка `res.partner`, писать свой
  визард было бы прямым дублированием уже готовой функциональности Odoo.
- **"Надёжность"**: полноценная разбивка по срывам сроков не
  переписывается заново на вкладке партнёра - краткая сводка + переход к
  уже существующему отчёту `vendor.delay.report`, чтобы не поддерживать
  два места с одной и той же логикой.
