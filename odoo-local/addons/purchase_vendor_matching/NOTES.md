# Заметки по реализации

Требование по п. 1.1 ТЗ: фиксировать здесь всё, что помечено `[Стандарт]`,
но потребовало кода, и любые отклонения от буквального текста ТЗ, с
объяснением почему. Весь этот модуль помечен `[Python]` в самом ТЗ, явных
несовпадений со «стандартом» нет — но есть одно место, где реализация
сознательно не покрывает пункт ТЗ буквально.

## Пара ИНН+КПП (п. 9.2 ТЗ) реализована не буквально

ТЗ: «если у обеих записей заполнен КПП — совпадение по паре ИНН + КПП».

Схема данных, которую отдаёт ИИ-экстракция счёта (`VENDOR_SCHEMA` в
`purchase_pdf_import/wizard/pdf_import_wizard.py`), **не содержит КПП
вообще** — только `name`, `tax_id`, `street`, `city`, `zip`, `country_code`,
`phone`, `email`. То же для УПД-мастера: `_find_or_create_vendor(name, vat)`
получает только ИНН, без КПП. Взять входящий КПП из документа физически
неоткуда без изменения схемы ИИ-извлечения, а это уже за рамками этого
небольшого изолированного модуля (и потребовало бы менять промпт и схему в
базовом модуле, что противоречит п. 0.1 ТЗ про "ни один модуль не изменяет
файлы базового").

Реализовано так: если поиск по ИНН находит **больше одной** карточки
(два филиала одного юрлица с разными КПП — единственный реальный случай,
когда это может произойти), метод `_match_vendor_by_inn` не выбирает
наугад ни одну из них, а возвращает "не найдено", и вызывающий откатывается
на прежний поиск по имени. Это **не сливает** разные карточки (ровно то, что
требует тест из п. 9.4 — «Одинаковый ИНН + разный КПП → две разные карточки,
не сливаются»), но и не выбирает правильный филиал автоматически — этого и
не может сделать код, не имеющий доступа к КПП из документа.

На проде на 22.07.2026 КПП не заполнен ни у одного контрагента (см.
п. 11.3 ТЗ), поэтому многозначность по ИНН сейчас в принципе невозможна —
это чисто защитный код на случай, если КПП начнут заполнять и заведут
несколько карточек с одним ИНН.

Если позже понадобится настоящее автоматическое различение по КПП — нужно
сначала научить `VENDOR_SCHEMA`/промпт ИИ-извлечения возвращать КПП
продавца (изменение в `purchase_pdf_import`, а не в этом модуле).
