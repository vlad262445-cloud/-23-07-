# purchase_vendor_matching

Модуль 8 из `TZ_purchase_modules.md` (раздел 9), первый в порядке внедрения
(п. 11.1 ТЗ) — база сейчас чистая (0 дублей поставщиков), и каждый
импортированный до этой правки счёт был шансом получить дубль.

## Что делает

Мастера ИИ-импорта (`purchase.pdf.import.wizard` и `purchase.updd.import.wizard`)
искали существующего поставщика **только по названию**. Расхождение в
написании («ООО Дата-В» / «ООО "Дата-В"» / лишний пробел) создавало вторую
карточку того же контрагента.

Модуль не трогает файлы `purchase_pdf_import` — только `_inherit` обоих
мастеров, override `_find_or_create_vendor`:

1. Ищем по **нормализованному ИНН** (только цифры), но лишь если ИНН
   проходит проверку контрольной суммы (алгоритм ФНС для 10 цифр — юрлицо,
   12 цифр — ИП).
2. Если по ИНН находится больше одной карточки (разные филиалы одного
   юрлица с разными КПП) — не выбираем наугад, откатываемся на поиск по
   имени. См. `NOTES.md`, почему пара ИНН+КПП не реализована буквально.
3. Иначе — прежнее поведение без изменений: имя + `supplier_rank > 0`,
   затем имя без этого условия, затем создание новой карточки.

Если поставщик найден по ИНН, а название в счёте отличается от названия в
карточке — заказ (или, для УПД без заказа, приёмка) всё равно получает
этого поставщика (ИНН авторитетнее), но в чат прилетает примечание с обоими
названиями, чтобы это не осталось незамеченным.

ИНН нормализуется (только цифры) **перед записью** — и в новых карточках, и
при дозаполнении существующих. `post_init_hook` разово нормализует ИНН у уже
существующих поставщиков при установке модуля (идемпотентно).

Ручной выбор поставщика (поле «Поставщик» на мастере PDF-импорта) по-прежнему
имеет приоритет над любым автопоиском — это поведение не менялось.

## Как проверить руками

1. Установить модуль на копию боевой базы.
2. Взять PDF-счёт от уже существующего поставщика (например, ООО "АДАР" —
   ИНН `7806513674`), но с названием, написанным иначе, чем в карточке.
   Оформить заказ через «Импортировать».
3. Убедиться, что **не** создалась вторая карточка контрагента, и что в
   чат заказа добавилось примечание «Поставщик определён по ИНН…».
4. Оформить счёт от совершенно нового поставщика — карточка должна
   создаться с ИНН без пробелов/букв, только цифры.
5. То же для УПД без привязанного заказа («Подтвердить» на
   `purchase.updd.import.wizard`, когда `purchase_order_id` не заполнен) —
   примечание должно попасть в чат созданной приёмки.

## Тесты

- `tests/test_inn_utils.py` — контрольная сумма ИНН на реальных ИНН из
  боевой базы (16 юрлиц + 6 ИП на 22.07.2026), включая один реальный ИНН
  с испорченной контрольной суммой, уже присутствующий в проде.
- `tests/test_vendor_matching.py` — поиск по ИНН для обоих мастеров,
  откат на поиск по имени при невалидном/неоднозначном ИНН, дозаполнение
  только пустых полей, нормализация при создании, примечание в чат (в т.ч.
  сквозной тест `action_import` целиком с замоканным вызовом ИИ).

## Приёмка (п. 9.5 ТЗ)

Юнит-тесты не заменяют один прогон: реальные PDF-счета от нескольких уже
существующих поставщиков нужно прогнать через мастер и убедиться, что
новых карточек не появилось ни одной.
