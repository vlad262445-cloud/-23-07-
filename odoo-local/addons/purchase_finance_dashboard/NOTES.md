# Заметки по реализации

## `cost_plan_root_id` - не потребовался рекурсивный CTE

П. 7.4 ТЗ явно предлагает считать корневой план рекурсивным CTE в SQL-view.
Перед тем как писать его, проверено на живом инстансе (`odoo shell`):
`account.analytic.account` в этой версии Odoo уже имеет **нативное stored
поле `root_plan_id`** (`_compute_related` до корня по `plan_id.parent_id`).
Проверено на реальных данных: `1.121 Компл. Приклады` (план `1.12
Комплектующие`, сам являющийся подпланом `1. Производственные`) →
`root_plan_id` = `1. Производственные`; плоская `1.20 Инструмент режущий` →
туда же. Используется напрямую (`caa.root_plan_id`) - никакого
собственного рекурсивного SQL. Дублировать логику, которую Odoo уже считает
сам, было бы лишним риском разойтись с ней при будущем обновлении
`account_analytic`. См. правило проекта - проверять реальный инстанс, а не
угадывать по документации/памяти о версии (тот же принцип, что уже стоил
времени в `purchase_registry_ux` с цветовой палитрой и виджетом "date").

## `is_received` - через `stock_picking`, не `request_state`

П. 7.1 ТЗ прямо предупреждает не повторять баг "сравнение request_state ==
'in_stock' даёт ложный ноль на маршрутах post_payment/split_50_50, где
request_state после оплаты уходит в invoice_paid" - тот же класс проблемы,
что уже был найден и описан в комментариях `purchase_pdf_import`
(`_request_state_reached`). SQL-view проверяет факт `stock_picking` со
`state = 'done'` и `picking_type.code = 'incoming'`, связанного через уже
существующее stored-поле `source_purchase_order_id` (см.
`purchase_pdf_import/models/stock_picking.py`) - никакой новой связи через
`group_id` заводить не пришлось. Регрессионный тест -
`test_is_received_uses_stock_picking_not_request_state_regression`.

## `days_frozen`/`days_debt` - через `create_date`/`date_done`, не через даты из документов

"Дней с даты первого платежа" по ТЗ - дата первого платежа технически есть
только в OCR-распознанном тексте (`purchase.payment.line.payment_date` -
`Char`, "Дата платежа (по документу)", та же проблема уже описана в
`purchase_finance_workspace/NOTES.md` для группировки "Оплачено за период").
Использован `min(purchase_payment_line.create_date)` - момент, когда
платёж реально ЗАРЕГИСТРИРОВАН в системе, тот же принцип, что и у самого
`amount_paid` (тоже строится по этой же таблице). Для `days_debt` -
`stock_picking.date_done` (реальное поле подтверждённой приёмки, не текст).
Оба - разумное приближение, не точная дата по документу; если заказчику
понадобится точность день-в-день по факту - нужно отдельное Date-поле на
`purchase.payment.line`, вне рамок этого ТЗ (тот же вывод уже сделан в
модуле 6/`purchase_finance_workspace`).

## `days_in_stage` - новый механизм, не в самой TZ буквально

Ни на `purchase.order`, ни где-либо ещё не было истории смены
`lifecycle_stage` (само поле - compute+store БЕЗ tracking=True). Добавлены
`lifecycle_stage_marker`/`lifecycle_stage_since` (`models/purchase_order.py`,
этот же модуль) - "sticky"-приём, но применённый к полю, которое может
менять значение много раз за жизнь заказа (в отличие от однократных
`completed_date`/`approval_date` в других модулях): при каждом пересчёте
сравнивается новое значение `lifecycle_stage` с прошлым сохранённым
(`lifecycle_stage_marker`), и `lifecycle_stage_since` обновляется только
при реальном переходе. Предыдущее значение читается ОДНИМ SQL-запросом
напрямую из базы, а не через `order.lifecycle_stage_marker` внутри
собственного же compute - чтение поля, которое в тот же момент вычисляется
этим же методом, в Odoo небезопасно (риск реентерабельности/устаревшего
кэша). И `lifecycle_stage`, и `lifecycle_stage_since` - мягкая зависимость
от `purchase_registry_ux` (первое поле физически принадлежит тому модулю);
если он не установлен, `days_in_stage` везде 0.

## Что сознательно НЕ реализовано

- **`spreadsheet.dashboard` сводные страницы** (п. 7.3, пункт 4 "Сводная
  страница"). Формат записи - JSON-блоб конкретной версии `o-spreadsheet",
  собрать его вручную "вслепую" (без интерактивного редактора, которым
  обычно и создают такие страницы через "Вставить в таблицу" из
  pivot/graph) - source of new, непроверяемых визуально багов, а не
  экономия времени: та же ситуация, что уже подробно описана в
  `feedback-odoo-ui-workflow` (память проекта) - без реальной визуальной
  проверки в браузере пользователя оценить корректность такого JSON
  невозможно. Все данные для этих страниц уже готовы (та же модель
  `purchase.finance.report`, те же pivot/graph экраны) - пользователь может
  собрать сводную страницу через штатный UI Odoo ("Вставить в таблицу" на
  любом из экранов этого модуля) за пару кликов, когда/если понадобится.
- **"Дополнительно предлагаемые показатели"** (в тексте ТЗ каждый такой
  пункт прямо помечен как "не было в постановке, но ценно") - средний срок
  "деньги → товар", оборот по отделам, средний чек, доля предоплаты, доля
  заказов без отклонений, сквозной SLA, заявки без заказа старше 7 дней,
  заказы без ИИ-распознавания, частота позиций/разброс цен между
  поставщиками. Все ОБЯЗАТЕЛЬНЫЕ показатели и экраны (п. 7.1-7.9, без
  пометки "дополнительно") реализованы полностью - см. README.md. Эти
  пункты сами по себе большие подзадачи (некоторые требуют новых полей),
  и ТЗ прямо не требует их для приёмки модуля.
- **"Требуется от меня"** (закупщик, п. 7.8) переиспользует
  `activity_ids.user_id = uid` напрямую новым действием, а не ищет и не
  переиспользует некий уже существующий в реестре фильтр (в реестре
  закупок такого готового фильтра с этим именно доменом не нашлось) -
  минимальное новое действие вместо переписывания реестра.
- **"Нет УПД по принятому"** (закупщик) сравнивает `request_state ==
  'in_stock'` напрямую, а не "дошли до in_stock или дальше"
  (`_request_state_reached`, приватный метод `purchase.order`, не
  предназначенный для использования в domain `ir.actions.act_window`) -
  тот же класс приближения, что уже принят в других местах проекта.

## Тест "не падает на пустой базе" - без удаления реальных заказов

Восстановленная база уже содержит ~32 реальных заказа. Чтобы не рисковать
каскадными побочными эффектами удаления реальных записей ради теста (даже
внутри отката транзакции `TransactionCase`), пустой результат проверяется
доменом, заведомо не находящим ни одной строки (`order_id = 0`), а не
физической очисткой таблицы - тот же практический результат (SQL-view на
пустом наборе строк не должна падать), без риска.
