# purchase_finance_dashboard

Модуль из раздела 7 ТЗ, седьмой и последний в основной очереди внедрения
(п. 11.1: "самый крупный модуль - делать последним, после 1-5" - здесь
после 1-6). Финансовые дашборды под 4 роли.

## Ядро: `purchase.finance.report`

Read-only SQL-view (`_auto = False`), одна строка на заказ. Бесплатно даёт
pivot/graph/list/search/group_by/XLSX/"Вставить в таблицу". Поля и формулы
сумм - см. `models/purchase_finance_report.py` (сохранены дословно по
п. 7.1 ТЗ) и `NOTES.md` (обоснование решений, не буквально прописанных в
ТЗ - `cost_plan_root_id` через нативное поле Odoo вместо CTE, `days_frozen`/
`days_debt`/`days_in_stage`).

Мягкие зависимости (модуль ставится и работает без 3-5, п. 11.1 ТЗ):
`lifecycle_stage` (из `purchase_registry_ux`) - `NULL`, если не установлен.

## Новые роли (п. 7.2)

`group_ceo`/`group_owner` - только чтение, наследуют `group_observer`
(видимость всех заявок/заказов без единого нового `ir.rule`). Известные
носители (`fedchin@odoo` → Собственник, `gibieva@odoo` → ГД) проставляются
`post_init_hook`, безопасно при отсутствии таких логинов. Галочки
`is_ceo`/`is_owner` добавлены в «Роли сотрудников» рядом с уже
существующими пятью.

**Главный закупщик видит все 4 дашборда** (требование заказчика от
22.07.2026) - его группа стоит на каждом верхнем меню, доступ только на
чтение, ни одного нового права на запись не добавлено ни для одной роли -
см. `security/`.

## Обязательный набор экранов по затратам (п. 7.4) - собран один раз

Строится один раз на общих list/kanban/graph/pivot вьюхах модели (context
задаёт measure/groupby под каждый экран - см. `views/
purchase_finance_report_views.xml`), подключается в разные меню по
матрице ТЗ:

1. Затраты по верхнему уровню (graph) - Производственные/Общефирменные/Инвестбюджет.
2. Затраты по статьям × период (pivot, с раскрытием от корня плана).
3. Затраты по категориям × период (pivot).
4. Статья × Категория (pivot).
5. Статья × Поставщик (pivot).
6. Статья × Отдел (pivot).
7. Динамика статьи (graph line).
8. Заморозка и долг в разрезе статей (pivot).

Плюс общая плитка «Без аналитики» (заказы без статьи или категории).

## Дашборд «Собственник» (п. 7.5)

Меню «Дашборд · Собственник»: заморозка по поставщикам, заморозка по
возрасту (интервалы 0-7/8-14/15-30/31-60/60+), долги по поставщикам,
динамика по периодам, концентрация на поставщиках, ВСЕ 8 экранов затрат,
без аналитики.

## Дашборд «Генеральный директор» (п. 7.6)

Меню «Дашборд · Генеральный директор»: воронка закупок (graph по
`lifecycle_stage`), возраст на этапе (список, `days_in_stage`, подсветка
>5 дней), «Согласование: кто держит» (pivot/список на
`purchase.approval.line`, новое поле `days_to_decide`), отделы, экраны
затрат 1/2/3/4/6/7/8 (без 5 - см. матрицу ТЗ), «Поставщики: срыв сроков»
(переиспользована стандартная `vendor.delay.report` из `purchase_stock` -
своя модель не писалась), без аналитики.

## Дашборд «Финансист» (п. 7.7)

Меню «Дашборд · Финансы» - дополняет `purchase_finance_workspace` (там
«что делать сейчас», здесь «как обстоят дела»), не дублирует его: к
оплате, оплаты по месяцам × поставщик, затраты 2/7/8 с мерами
`amount_paid`/`amount_residual` вместо `amount_total`, документы не в
порядке, без аналитики.

## Дашборд «Закупщик» (п. 7.8)

Меню «Дашборд · Закупки»: требуется от меня, мои заказы по этапам, заявки
без заказа, просрочен желаемый срок, в пути, отклонённые, нет УПД по
принятому, поставщики: срыв сроков, статья × поставщик (основной, только
свои закупки), по статьям/категориям × период (свои).

## Что не сделано и почему

См. `NOTES.md` - сводные страницы `spreadsheet.dashboard` и
«дополнительно предлагаемые» показатели сознательно отложены (все
обязательные пункты реализованы).

## Тесты

`tests/test_finance_report.py` - суммы (заморозка/долг/ни то ни другое),
регрессия is_received через stock_picking, `cost_plan_root_id` для любой
глубины вложенности + отсутствие потерь/задвоения при суммировании, заказ
без категории не ломает pivot, пустой result set не падает,
`lifecycle_stage_since` меняется только при реальном переходе, права
доступа (ceo/owner - чтение без записи, chief_buyer - все 4 дашборда без
новых прав записи).
