# Заметки по реализации

## `res.users.department_id` - конфликт имён с штатным `hr`, не мелочь

Обнаружено при первой же попытке установки: `hr` заводит своё
`res.users.department_id` (`related='employee_id.department_id'`, на
`hr.employee.department_id`, то есть `hr.department`) - тем же именем,
что уже несколько месяцев использует `purchase_pdf_import`
(`Many2one('purchase.request.department')`, "домашний отдел" для
ограничения заявок). Проверено `env['res.users']._fields['department_id']`
после установки `hr`: `comodel_name` = `purchase.request.department` (от
purchase_pdf_import), но `related` = `employee_id.department_id` и
`store` = `False` (от hr) - Odoo смержила атрибуты ДВУХ разных
определений в одно поле вместо ошибки при загрузке модуля. Падает не
сразу, а при первом обращении на запись (`ValueError: Wrong value for
res.users.department_id: hr.department()`) - т.е. в проде это всплыло бы
не при установке, а при первом же сохранении пользователя с "домашним
отделом", что хуже.

Одноимённые поля с разным `comodel_name` на одной модели через два разных
`_inherit` - это не "то, что можно настроить по-другому", а комбинация,
которую сама Odoo не поддерживает корректно. Единственный надёжный выход -
переименование.

**Исправлено** (с явного согласия пользователя после того, как ему был
показан диф - см. чат): `purchase_pdf_import.models.res_users.py` -
`department_id` → `purchase_department_id`. Правки везде, где
использовалось (`purchase_request.py`: default заявки и
`_check_department_matches_user`; `res_users_views.xml`). Старые 5
значений (Киселев/Кустанович/Мицуков/Наточисел/Чутчиков) переносятся
сырым SQL в `hr_workwear.__init__._migrate_res_users_department_column` -
**в hr_workwear, не в purchase_pdf_import** - именно этот модуль вводит
зависимость от `hr` и обнажает конфликт; `purchase_pdf_import` сам по
себе (без `hr`) в этой правке не нуждался бы. Миграция читает старую
физическую колонку `department_id` в таблице `res_users` напрямую (Odoo
никогда не дропает старые колонки сама - после переименования поля она
просто становится "осиротевшей", данные в ней остаются нетронутыми) -
работает независимо от того, что там сейчас думает ORM про это имя поля.
Порядок в `post_init_hook` важен: миграция - первым шagom, до правок
справочника А (иначе `natochisel.purchase_department_id` ещё пуст на
момент проверки "было ли Продажи").

## `purchase_department_id` (hr.employee) - тот же класс небуквального "related", что и в модуле 8

ТЗ п. 10.2 описывает связь `hr.employee` → `purchase.request.department`
как обычный `related`. Технически это обратный поиск (сопоставление
хранится НА `purchase.request.department` через `hr_department_id`,
указывающий на `hr.department` - а нужно от `hr.employee.department_id`
дойти до записи `purchase.request.department`, которая на неё ссылается).
Штатный Odoo `related` не умеет обратных путей. Реализовано обычным
`compute` без `store` (батч-поиском по всему recordset одним
`search_read`) - тот же приём и то же обоснование (нет надёжного триггера
на "запись в ДРУГОЙ модели поменялась"), что уже описано для
`purchase_department_id` в `purchase_vendor_card/NOTES.md`.

То же самое для пяти "колонок-размеров" (`workwear_size_robe` и т.п.) на
экране "Личный состав" - п. 10.7 ТЗ тоже говорит "через related", но
Odoo related не умеет фильтровать One2many по конкретному типу СИЗ.
Обычный compute поверх уже загруженного (prefetch) `workwear_size_ids` -
лишних запросов не добавляет, т.к. набор размеров на человека всегда
маленький (по одной записи на тип).

## "Потребность" - обычная модель с `_refresh()`, не SQL-view

Модуль 6/7 для похожих отчётов используют `_auto = False` + `init()` с
сырым `CREATE VIEW`. Здесь это осознанно не сделано: "дата очередной
выдачи" считается как `COALESCE(последняя выдача по этому типу, дата
приёма) + период нормы` - арифметика с `relativedelta`, которую пришлось
бы либо продублировать в SQL (риск разойтись с версией на Python - тот же
класс риска, которого проект уже избегал раньше), либо усложнять SQL до
неоправданной степени. Вместо этого - обычная (`_auto=True`) модель
`hr.workwear.requirement.line`, полностью пересобираемая методом
`_refresh()` **поверх того же самого метода**
(`hr.employee._get_workwear_norm_due_lines`), что считает `workwear_
overdue_count`/`workwear_next_due_date` на карточке сотрудника - единый
источник арифметики дат, а не два. Плата - это снимок, не живой view:
обновляется ежедневным cron (тем же, что шлёт напоминания кладовщику) или
вручную кнопкой "Обновить потребность" в меню "Действие" списка.

## `workwear_overdue_count` - `search=` есть, сортировка - нет (тот же размен, что в модуле 8)

Нужно фильтровать "Просрочено" по количеству просроченных позиций - поле
не хранится (пересчитывается на лету, т.к. "просрочено" зависит от
текущей даты, а не только от данных в базе). Добавлен `search=
'_search_workwear_overdue_count'` тем же батч-приёмом, что
`vendor_debt_total` в `purchase_vendor_card` - фильтрация работает,
сортировка по этому полю - нет (Odoo умеет транслировать `search`, но не
`order`, для нехранимых полей). Список "Просрочено" сортируется по
`workwear_next_due_date` вместо этого - практичная замена, не то же
самое, но не требует хранения.

## `hr.workwear.norm._check_job_or_department` - `@api.constrains` не сработал бы сам на пустом create()

Найдено тестом `test_norm_requires_job_or_department_not_both`: если ни
`job_id`, ни `department_id` вообще не передать в `create()` (не False
явно, а просто не упомянуть ключ), `@api.constrains('job_id',
'department_id')` **не запускается** - Odoo привязывает проверку к полям,
реально тронутым в `vals`, а если оба ключа отсутствуют, ни один из двух
не "тронут". Проверено вручную: прямой вызов
`record._check_job_or_department()` кидает `ValidationError` как надо,
но при `create({'type_id': ..., 'period_months': ...})` без обоих полей
- проверка тихо не срабатывает и запись сохраняется без нормы вообще.
Исправлено явным вызовом `_check_job_or_department()` в переопределённом
`create()`, в дополнение к декоратору (который по-прежнему нужен для
`write()`-случая: кто-то очистит оба поля после создания).

## Порядок пользователь->роспись реестра - по логину, не по фуzzy-имени

TZ п. 10.3 предупреждает о риске "Ерёмин"/"Еремин" через ё/е при импорте
по ФИО - актуально для 53 человек в `data/hr_employee_import.csv`
(штатный импорт Odoo, сопоставление по имени). Для 13 сотрудников с
учётками Odoo сопоставление сделано НЕ через нормализацию имени, а через
точный `login` (`_KNOWN_LOGIN_ROSTER` в `__init__.py`) - логин уникален и
уже был вручную сверен построчно с реестром при написании модуля (см. эту
же переписку), риск ложного совпадения по имени здесь не возникает
вообще - используется более надёжный ключ там, где он есть.

Автоматическая проверка на дубли по нормализованному ФИО ("ё"→"е",
пробелы, регистр) прогнана один раз по всем 65 строкам реестра при
подготовке `data/hr_employee_import.csv` (скрипт, не входит в модуль) -
0 дублей, все имена уникальны после нормализации.

## Типы/размеры/сроки носки - разумные дефолты, не норма от заказчика

Стартовый набор `data/workwear_type_data.xml`/`workwear_size_data.xml`
(размерные сетки S-XXL, 44-64, 36-48 обувь, 54-62 головной убор,
`wear_period_months` по типу) - типовые отраслевые ориентиры для
редактирования в интерфейсе, не проверенная у заказчика норма. Реальные
нормы выдачи (`hr.workwear.norm`) намеренно оставлены пустыми - это
бизнес-решение заказчика (какая должность/отдел получает что и как
часто), придумывать его самостоятельно значило бы фабриковать
несогласованные бизнес-правила.
