# Заметки по реализации

## `finance_stage`: "approved, но ничего не подошло" → `wait_approval`

Раздел 6.1 ТЗ перечисляет 7 значений, но не покрывает явно случай
`approval_state == 'approved'`, маршрут `post_payment`, оплата ещё не
началась (деньги по этому маршруту идут ПОСЛЕ получения - до самого факта
получения товара финансисту в принципе нечего делать). Ни один из
перечисленных ключей буквально не подходит («requires_prepayment» ложно,
«to_upload_slip» требует `_request_state_reached('invoice_paid')`, которого
ещё не было).

Решение: переиспользовать `wait_approval` как «в очереди финансиста сейчас
пусто» вместо восьмого значения под один этот случай - с точки зрения
бухгалтера оба состояния («ещё не согласовано» и «согласовано, но пока
объективно рано платить») означают одно и то же: «не мой ход прямо
сейчас». Подпись «Ждёт согласования» становится чуть неточной для второго
случая, но заводить отдельное значение ради одной формулировки казалось
избыточным - если заказчик увидит эту колонку в kanban и она смутит, это
подходящее место для маленькой правки текста, а не для новой ветки логики.

## `to_pay`/`to_pay_urgent` только для full_prepay/split_50_50

П. 6.1 ТЗ формулирует условие как "маршрут требует оплаты до отгрузки" -
это `payment_type in ('full_prepay', 'split_50_50')` (обе схемы платят
ДО получения, в отличие от `post_payment`). Вынесено в константу
`PREPAYMENT_ROUTES`, чтобы не терять смысл за булевым выражением на месте.

## Приоритет считается через множество совпадений, а не if/elif

`_finance_stage_matches()` возвращает МНОЖЕСТВО ключей, чьё условие сейчас
истинно (например, `to_pay` и `to_pay_urgent` одновременно) -
`_finance_stage_key()` берёт первый по `FINANCE_STAGE_PRIORITY`. Так порядок
приоритета - ровно то, что написано в ТЗ (`inn_mismatch → to_pay_urgent →
to_surcharge → to_pay → to_upload_slip → done → wait_approval`), лежит в
одном месте и покрыт отдельным тестом на сам порядок, а не размазан по
условиям ветвления.

## Поставщик (кратко) - мягкая зависимость от `purchase_registry_ux`

`finance_partner_display_name` проверяет `'short_name' in
self.env['res.partner']._fields` в РАНТАЙМЕ, а не в `@api.depends` (поле
может не существовать вовсе, если `purchase_registry_ux` не установлен - п.
11.1 ТЗ: "модули 6 и 7 используют поля модулей 3-5, но должны
устанавливаться и работать без них"). Тот же приём, что уже применён в
`purchase_registry_ux` для `purchase_order_archive.is_completed`.
Побочный эффект - если `short_name` меняется вручную уже ПОСЛЕ вычисления
`finance_partner_display_name`, пересчёт не подхватит это немедленно (поле
не в depends), только на следующий пересчёт по `partner_id.name`. Тот же
компромисс, что уже принят и задокументирован в `purchase_registry_ux`.

## "Оплачено за период": группировка по месяцу `date_order`, а не по дате платежа

На `purchase.order` нет отдельного поля "дата платежа" - `purchase.payment.
line.payment_date` существует, но это `Char` ("Дата платежа (по
документу)"), а не `Date`/`Datetime` - group_by/pivot по месяцам на текстовом
поле не работает. Используется `date_order` ("Крайний срок" по факту, дата
создания заказа) как ближайший доступный период - разумное приближение
для "оплачено за период", а не точная дата фактического платежа. Если
заказчику важна точность по факту оплаты - потребуется отдельное Date-поле
на `purchase.payment.line`, это уже выходит за рамки этого ТЗ.

## Pivot группируется по `partner_id`, не по `finance_partner_display_name`

`finance_partner_display_name` - compute БЕЗ `store=True` (принципиально:
чтобы `short_name` мог измениться и это сразу отражалось при следующем
открытии экрана, не тянуть за собой сложность хранимого поля ради
косметики). Group_by/pivot по нехранимому compute-полю Odoo не поддерживает
- в pivot строки группируются по `partner_id` (нативное, хранимое поле),
`finance_partner_display_name` используется только там, где нужен просто
вывод текста (kanban-карточка, список), не группировка.

## `payment_import_wizard` не тронут

Кнопка «Загрузить платёжку» во всех списках вызывает уже существующий
`purchase.order.action_open_payment_import_wizard()` как есть - ни новый
wizard, ни правка существующего (п. 6.4 ТЗ).
