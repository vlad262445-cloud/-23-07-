# purchase_finance_workspace

Модуль из раздела 6 ТЗ, шестой по очереди внедрения (п. 11.1) - отдельное
рабочее место финансиста (роль - существующая
`purchase_pdf_import.group_accountant`), чтобы не искать нужное в общем
реестре.

## Что делает

### `finance_stage` — вычисляемый маршрутизатор (п. 6.1)

| Ключ | Подпись | Условие |
|---|---|---|
| `inn_mismatch` | Сверить ИНН | есть `payment_line_ids` с `partner_matched = False` |
| `to_pay_urgent` | Срочная оплата | `to_pay` + `payment_priority in ('1','2')` |
| `to_surcharge` | Требуется доплата | `0 < amount_paid < amount_total * 0.95` |
| `to_pay` | Требуется оплата | согласовано, `amount_paid == 0`, маршрут платит до отгрузки |
| `to_upload_slip` | Загрузить платёжку | оплата прошла, но нет `payment_line_ids` или `payment_skipped` |
| `done` | Закрыто | оплачено полностью, платёжки на месте |
| `wait_approval` | Ждёт согласования | ещё не согласовано (или согласовано, но объективно рано - см. `NOTES.md`) |

Порядок в таблице - это и есть приоритет при нескольких совпадениях,
зафиксирован константой `FINANCE_STAGE_PRIORITY` и отдельным тестом.

### Экраны (п. 6.2, всё [Стандарт])

Новое приложение верхнего уровня **«Финансы»**
(`group_accountant`/`purchase.group_purchase_manager`) - «Реестр закупок»
никуда не делся, это дополнительное меню.

- **Рабочий стол** - kanban по `finance_stage` (бесплатно: колонки,
  счётчики, сворачивание пустых). Карточка: № закупки, поставщик
  (кратко), № счёта, сумма, остаток, приоритет, срок. Красная полоса слева
  для `to_pay_urgent`/`inn_mismatch`.
- **Требуется оплата / Срочная оплата / Требуется доплата / Загрузить
  платёжку / Сверить ИНН** - список с фиксированным доменом по
  `finance_stage`, урезанные колонки, кнопка «Загрузить платёжку» (вызывает
  существующий `action_open_payment_import_wizard`, новый wizard не
  писался).
- **Оплачено за период** - список + pivot (строки - поставщик, колонки -
  месяц `date_order`, мера - `amount_paid`).

### Поля-помощники (п. 6.3)

- `amount_residual_purchase` (Monetary, stored) = `amount_total - amount_paid`.
- `payment_due_date` (Date) - с первой связанной заявки (`desired_date`).
- `days_waiting_payment` (Integer, НЕ stored - зависит от "сегодня"). Для
  сортировки по времени ожидания использовать `approval_date` (stored,
  фиксируется один раз при первом переходе в `approved`, не сбрасывается).

### Что не сделано и почему

- `payment_import_wizard` не тронут (п. 6.4).
- Права доступа бухгалтера не менялись (п. 6.4).
- «Реестр закупок» не скрыт и не изменён (п. 6.4).

## Как проверить руками

1. Установить модуль на пользователе с ролью «Бухгалтер».
2. Открыть «Финансы» → «Рабочий стол» - kanban с колонками по этапам.
3. Проверить, что заказ с частичной оплатой (0 < оплачено < 95%) появляется
   в «Требуется доплата», а не в «Требуется оплата».
4. На заказе с несверенным платежом (`partner_matched = False`) - должен
   попасть в «Сверить ИНН» даже если по остальным признакам подходит и под
   другой этап (приоритет `inn_mismatch` выше всех).
5. Нажать «Загрузить платёжку» в любом списке - должен открыться тот же
   мастер, что и на форме заказа.
6. «Оплачено за период» - переключиться на pivot, развернуть по месяцам.
