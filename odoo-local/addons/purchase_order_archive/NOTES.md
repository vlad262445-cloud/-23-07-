# Заметки по реализации

## «domain=[], снимает ограничение» — как это реализовано технически

ТЗ (п. 4.2): «В search-view реестра — фильтр «Показать архивные»
(`domain=[]`, снимает ограничение)».

Рестрикция «не архивные по умолчанию» реализована **не** через
`domain=` на самом `ir.actions.act_window` — если бы ограничение сидело
там, оно было бы жёстким и НИКАКОЙ фильтр в панели поиска не смог бы его
снять (`domain` экшена и активные фильтры комбинируются через AND, а не
заменяют друг друга).

Вместо этого — два фильтра в одной группе (без `<separator/>` между ними):

```xml
<filter name="filter_hide_archived" domain="[('is_archived', '=', False)]"/>
<filter name="filter_show_archived" domain="[]"/>
```

`filter_hide_archived` включён по умолчанию через контекст экшена
(`search_default_filter_hide_archived: 1`). Когда пользователь ДОПОЛНИТЕЛЬНО
включает `filter_show_archived`, оба фильтра активны одновременно и лежат в
одной группе → Odoo объединяет их через `OR`, а `OR` с пустым доменом
поглощает всю группу целиком (пустой домен = «без ограничений»). Результат -
ограничение снято, видно всё, включая архивные, без необходимости уходить в
отдельное меню «Архив закупок».

## Колонка `is_completed` в реестре — временный фолбэк

П. 4.1 ТЗ: «is_completed встраивается в lifecycle_stage из модуля
`purchase_registry_ux` … если модуль не установлен - показывать отдельную
колонку `is_completed`». Зависимость в манифесте на `purchase_registry_ux`
намеренно не поставлена (модули должны внедряться независимо).

На момент реализации этого модуля `purchase_registry_ux` **ещё не
существует** (он идёт пятым в очереди работ, этот - четвёртым, см. п. 11.1
ТЗ), поэтому колонка добавлена безусловно - сейчас это просто единственный
рабочий вариант.

**Когда будет реализовываться `purchase_registry_ux`**, ему нужно будет
самому решить, как ужиться с этой колонкой (ТЗ явно перекладывает эту
развилку на try/except/условную загрузку, а не на жёсткую зависимость):
либо скрыть/заменить колонку `is_completed` из этого модуля своим xpath
(если `purchase_order_archive` установлен), либо просто читать поле
`is_completed` напрямую в своём compute `lifecycle_stage` (оно к тому
моменту уже физически существует на `purchase.order`, если этот модуль
установлен) - но тогда `purchase_registry_ux` должен аккуратно обработать
случай, когда `purchase_order_archive` НЕ установлен и поля `is_completed`
вовсе нет (через `'is_completed' in order._fields` или аналогичную
проверку), а не безусловно на него полагаться.
