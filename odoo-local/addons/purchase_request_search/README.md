# purchase_request_search

Модуль из раздела 3 ТЗ (`purchase_request_search`), второй по очереди
внедрения (п. 11.1) — маленький, полностью стандартный, не блокирует более
сложные модули ревью.

## Что делает

XML-наследование `purchase_pdf_import.view_purchase_request_search`
(`xpath`/`position`, файлы базового модуля не тронуты). Добавляет:

- поиск по позициям заявки (`purchase.request.line`) — по названию, коду
  товара, доп. характеристике и примечанию (`filter_domain` через
  `line_ids`, аналогично тому, как уже сделано в «Реестре закупок» через
  `order_line`, но по текстовым полям, а не по `product_id` — позиции
  заявки намеренно не привязаны к товарному каталогу);
- фильтр «Без заказа» (`purchase_order_id` не заполнен);
- фильтр «Просрочена желаемая дата» (`desired_date` в прошлом, заявка не
  в `in_stock`/`merged`);
- группировка «Заявитель» (`requested_by`).

Ни одного нового поля, модели или права доступа — только элементы
search-view, ничего не удаляется и не заменяется (только `position="after"`).

## Как проверить руками

1. Установить модуль.
2. Открыть «Заявки на закупку» — в панели поиска ввести часть названия
   товара из позиции любой заявки (`purchase.request.line.name`) — заявка
   должна найтись.
3. То же для кода товара/артикула и текста примечания.
4. Включить фильтр «Без заказа» — список должен показать только заявки без
   `purchase_order_id`.
5. Включить «Просрочена желаемая дата» на заявке с `desired_date` в
   прошлом и статусом не «На складе»/«Объединена» — должна попасть в список.
6. Сгруппировать по «Заявитель» — заявки должны разбиться по `requested_by`.

## Тесты

`tests/test_request_search_filters.py` — проверяет сами домены фильтров
(которые прописаны в `arch` search-view) напрямую через `search()`, без
эмуляции клика по кнопке поиска в UI.
